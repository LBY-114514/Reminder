# 挑战杯问题记录本地提醒程序设计

## 背景

用户希望为后续参加挑战杯准备一个本地工具：遇到问题时能快速记录，并在设定时间由电脑提醒自己查看。工具面向个人使用，不需要微信推送、不需要联网、不需要复杂账号系统。

## 目标

- 提供一个本地可启动的小程序。
- 用户通过桌面快捷方式或启动脚本打开。
- 支持新增、查看、编辑、删除挑战杯问题记录。
- 每条问题可设置独立提醒时间。
- 数据保存在当前项目文件夹内，便于备份和迁移。
- 程序运行时，到点通过电脑通知或程序内提示提醒用户。

## 非目标

- 不做微信提醒。
- 不做多人协作。
- 不做云同步。
- 不做复杂权限系统。
- 第一版不追求精美 UI，先保证功能完整；UI 后续可单独美化。

## 推荐方案

采用 **Python 本地服务 + 浏览器页面**：

- `app.py` 负责启动本地服务、读写数据、检查提醒。
- `web/` 存放前端页面、样式和交互脚本。
- `data/issues.json` 存放问题记录，位于当前项目文件夹内。
- `start.bat` 用于一键启动，后续可给它创建桌面快捷方式。
- 启动后自动打开本地浏览器页面，例如 `http://localhost:8787`。

选择该方案的原因：

- 比正式桌面 App 更轻量，工作量适中。
- 比纯 HTML 网页更可靠，因为 Python 后台可以负责本地文件读写和提醒检查。
- 数据结构清晰，后续扩展 UI 或打包成桌面应用都方便。

## 用户流程

### 启动

1. 用户双击 `start.bat` 或桌面快捷方式。
2. 程序激活 `forskills` Conda 环境后启动 Python 服务。
3. 浏览器自动打开本地页面。
4. 程序开始定时检查未完成问题的提醒时间。

### 记录问题

1. 用户输入问题标题。
2. 可选填写详情。
3. 设置提醒时间。
4. 点击保存。
5. 记录写入 `data/issues.json`。

### 到点提醒

1. 后台检查发现某条待处理问题到达提醒时间。
2. 优先尝试发送 Windows 通知。
3. 如果系统通知不可用，则在程序页面显示明显提醒。
4. 用户可将问题标记为已处理，或修改提醒时间。

## 数据模型

`data/issues.json` 保存一个数组，每条记录包含：

```json
{
  "id": "唯一 ID",
  "title": "问题标题",
  "detail": "详细说明",
  "remind_at": "2026-05-10T20:30:00+08:00",
  "status": "pending",
  "created_at": "2026-05-10T19:00:00+08:00",
  "updated_at": "2026-05-10T19:00:00+08:00",
  "notified": false
}
```

状态值：

- `pending`：待处理。
- `done`：已处理。

提醒规则：

- 只提醒 `status = pending` 且 `notified = false` 的记录。
- 到点提醒成功或触发兜底提醒后，将 `notified` 设为 `true`，避免重复刷屏。
- 如果用户修改提醒时间，则重置 `notified = false`。

## 页面功能

第一版页面包含：

- 新增问题表单：标题、详情、提醒时间。
- 问题列表：显示标题、提醒时间、状态。
- 编辑功能：修改标题、详情、提醒时间。
- 删除功能：删除不需要的问题。
- 标记已处理：把问题状态改为 `done`。
- 过滤：查看全部、待处理、已处理。
- 到点提醒区域：当程序内提醒触发时展示醒目提示。

UI 第一版以清晰可用为主，不在本轮追求精美视觉。

## 接口设计

本地服务提供 JSON API：

- `GET /api/issues`：获取全部问题。
- `POST /api/issues`：新增问题。
- `PUT /api/issues/{id}`：更新问题。
- `DELETE /api/issues/{id}`：删除问题。
- `POST /api/issues/{id}/done`：标记已处理。
- `GET /api/reminders/due`：页面轮询到点提醒，用作通知兜底。

## 错误处理

- 如果 `data/issues.json` 不存在，程序自动创建空数组文件。
- 如果 JSON 文件损坏，程序保留损坏文件副本，并新建空数据文件，避免启动失败。
- 如果提醒时间格式不合法，接口返回错误，页面提示用户修改。
- 如果 Windows 通知发送失败，页面内提醒继续可用。
- 如果端口被占用，程序尝试使用备用端口并打开对应页面。

## 测试策略

按 TDD 实现核心逻辑：

- 数据文件不存在时能自动创建。
- 新增问题会写入 JSON。
- 更新提醒时间会重置 `notified`。
- 已处理问题不会触发提醒。
- 到点且未通知的问题会被识别为待提醒。
- 损坏 JSON 会被备份并恢复为空数据。

前端第一版可通过手动验证：

- 能新增、编辑、删除问题。
- 刷新页面后数据仍存在。
- 设置一个近时间提醒，程序运行时能触发提醒。

## 实施边界

第一版只实现本地单机版本。桌面快捷方式可以通过 `start.bat` 创建，不强制打包成 `.exe`。后续如果需要更像正式 App，可再考虑打包或 Electron。

## 自检结果

- 无未定需求或占位项。
- 方案与用户确认一致：本地程序、手动启动、数据放当前文件夹、电脑提醒。
- 范围可控，适合单次实现计划。
- “程序关闭后不提醒”已隐含在手动启动模型中：只有程序运行时才检查提醒。
